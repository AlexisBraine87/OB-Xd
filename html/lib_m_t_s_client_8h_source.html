17 /*
18 Steps for using the MTS-ESP client API to add microtuning support to a plug-in.
19 Steps 1 and 2 are required, however it is recommended to include further steps when
20 integrating:
21
22
23 1. REQUIRED: Register and de-register a plug-in instance as a client with MTS-ESP.
24 On startup in the plug-in constructor call:
25
26 MTSClient *client = MTS_RegisterClient();
27
28 Store the returned MTSClient pointer to supply when calling other MTS-ESP client API
29 functions. On shutdown in the plug-in destructor call:
30
31 MTS_DeregisterClient(client);
32
33
34 2. REQUIRED: Query retuning when a note-on message is received and adjust tuning accordingly.
35 When given a note call:
36
37 double freq = MTS_NoteToFrequency(client, midinote, midichannel);
38 OR
39 double retune_semitones = MTS_RetuningInSemitones(client, midinote, midichannel);
40 OR
41 double retune_ratio = MTS_RetuningAsRatio(client, midinote, midichannel);
42
43 MIDI channel arguments should use the range [0,15] however if you don’t know the MIDI
44 channel, use -1 (see step 6 for more on MIDI channels).
45
46
47 3. RECOMMENDED: Continuously query retuning whilst a note is held, allowing tuning to change
48 along the flight of a note. Do this if you can and as often as possible, ideally at the same
49 time as processing any other pitch modulation sources (envelopes, MIDI controllers, LFOs etc.).
50
51
52 4. RECOMMENDED: Provide an option to the user to select whether tuning is queried at note-on
53 only, as in step 2, or continuously, as in step 3. There are creative and practical
54 advantages to both, depending on the use case, and offering an option to the user will
55 provide the most useful MTS-ESP integration. If not offering such an option, continuous
56 retuning should be preferred over note-on only retuning.
57
58
59 5. RECOMMENDED: Query whether a note should be sounded when a note-on message is received.
60 The Scala .kbm keyboard mapping format allows for MIDI keys to be unmapped i.e. no frequency
61 is specified for them, and the MTS-ESP library supports this too. You can query whether a note
62 is unmapped and should be ignored with:
63
64 bool should_ignore_note = MTS_ShouldFilterNote(client, midinote, midichannel);
65
66 If this returns true, ignore the note-on and don’t play anything. Calling this function is
67 recommended but optional and a valid value for frequency/retuning will be returned for an
68 unmapped note. MIDI channel arguments should use the range [0,15] however if you don’t
69 know the MIDI channel, use -1.
70
71
72 6. RECOMMENDED: Always supply a MIDI channel when querying retuning or note filtering. Doing
73 so allows your plug-in to multi-channel tuning tables, useful for microtonal MIDI controllers
74 with more than 128 keys or working with large scales. Even if multi-channel tables are not
75 in use, a master may still make use of channel-specific note filtering for functions such as
76 key switches to change tunings. If your plug-in supports MPE and has a switch for enabling MPE
77 support, it is recommended to NOT supply a MIDI channel if MPE is enabled.
78
79
80 7. RECOMMENDED: If you are adding MTS-ESP support to a plug-in that already has some kind
81 of microtuning support, e.g. loading .scl or .tun files, let the local tuning automatically
82 override MTS-ESP, or provide an option for MTS-ESP retuning to be explicitly disabled.
83 This affords a user the option to use a different tuning to the global MTS-ESP table
84 for a specific plug-in instance.
85
86
87 8. OPTIONAL: Add support for MIDI Tuning Standard (or MTS, from the MIDI specification) SysEx
88 messages to your plug-in. When not connected to an MTS-ESP master plug-in, these can be used
89 to retune it instead, providing microtuing support even when MTS-ESP is not in use.
90 When a SysEx message is received, call:
91
92 MTS_ParseMIDIData(client, buffer, len); // if buffer is signed char *
93 OR
94 MTS_ParseMIDIDataU(client, buffer, len); // if buffer is unsigned char *
95
96 These will update a local tuning table which is used when querying retuning as in steps 2
97 and 3.
98
99
100 9. OPTIONAL: If you want to display to the user whether the plug-in is "connnected" to an
101 MTS-ESP master plug-in, call:
102
103 bool has_master = MTS_HasMaster(client);
104
105
106 10: OPTIONAL: It is possible to query the name of the current scale. This function is necessarily
107 supplied for the case where a client is sending MTS SysEx messages, however it can be used
108 to display the current scale name to the user on your UI too:
109
110 const char *name = MTS_GetScaleName(client);
111
112
113 11: EXTRAS: Helper functions are available which return the MIDI note whose pitch is nearest
114 a given frequency. The MIDI note returned is guaranteed to be mapped. If you intend to
115 generate a note-on message using the returned note number, you may already know which MIDI
116 channel it will be sent on, in which case you must specify this in the call, else the client
117 library can prescribe a channel for you. This is done so that multi-channel mapping
118 and note filtering can be respected. See below for further details.
119 */

121 // Opaque datatype for MTSClient.
122 typedef struct MTSClient MTSClient;

124 // Register/deregister as a client. Call from the plugin constructor and destructor.
125 extern MTSClient *MTS_RegisterClient();
126 extern void MTS_DeregisterClient(MTSClient *client);
127
128 // Check if the client is currently connected to a master plugin.
129 extern bool MTS_HasMaster(MTSClient *client);

131 // Returns true if note should not be played. MIDI channel argument should be included if possible (0-15), else set to -1.
132 extern bool MTS_ShouldFilterNote(MTSClient *client, char midinote, char midichannel);
133
134 // Retuning a midi note. Pick the version that makes your life easiest! MIDI channel argument should be included if possible (0-15), else set to -1.
135 extern double MTS_NoteToFrequency(MTSClient *client, char midinote, char midichannel);
136 extern double MTS_RetuningInSemitones(MTSClient *client, char midinote, char midichannel);
137 extern double MTS_RetuningAsRatio(MTSClient *client, char midinote, char midichannel);
138
139 // MTS_FrequencyToNote() is a helper function returning the note number whose pitch is closest to the supplied frequency. Two versions are provided:
140 // The first is for the simplest case: supply a frequency and get a note number back.
141 // If you intend to use the returned note number to generate a note-on message on a specific, pre-determined MIDI channel, set the midichannel argument to the destination channel (0-15), else set to -1.
142 // If a MIDI channel is supplied, the corresponding multi-channel tuning table will be queried if in use, else multi-channel tables are ignored.
143 extern char MTS_FrequencyToNote(MTSClient *client, double freq, char midichannel);
144 // Use the second version if you intend to use the returned note number to generate a note-on message and where you have the possibility to send it on any MIDI channel.
145 // The midichannel argument is a pointer to a char which will receive the MIDI channel on which the note message should be sent (0-15).
146 // Multi-channel tuning tables are queried if in use.
147 extern char MTS_FrequencyToNoteAndChannel(MTSClient *client, double freq, char *midichannel);
148
149 // Returns the name of the current scale.
150 extern const char *MTS_GetScaleName(MTSClient *client);
151
152 // Parse incoming MIDI data to update local retuning. All formats of MTS sysex message accepted.
153 extern void MTS_ParseMIDIDataU(MTSClient *client, const unsigned char *buffer, int len);
154 extern void MTS_ParseMIDIData(MTSClient *client, const char *buffer, int len);
